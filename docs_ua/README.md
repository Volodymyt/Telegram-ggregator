# Документація

Цей каталог є точкою входу до внутрішньої документації проєкту.

## Структура

- `project/` зберігає стабільні знання про проєкт, які змінюються нечасто.
- `planning/` зберігає дорожню карту, беклог, активні робочі елементи й архівні плани.
- `roles/` зберігає визначення ролей, межі відповідальності та очікування щодо співпраці.
- `adr/` зберігає архітектурні та процесні рішення, які мають залишатися легко доступними.

## Правила розміщення

- Довготривалий продуктовий і технічний контекст розміщуй у `project/`.
- Артефакти операційного планування розміщуй у `planning/`.
- Моделі відповідальності та межі ухвалення рішень розміщуй у `roles/`.
- Важливі рішення разом з обґрунтуванням розміщуй у `adr/`.

## Основні документи

- Починай з [`project/overview.md`](project/overview.md), щоб зрозуміти продуктовий намір і поточний контекст репозиторію.
- Використовуй [`project/requirements.md`](project/requirements.md) як канонічну специфікацію MVP.
- Використовуй [`project/architecture.md`](project/architecture.md) для короткого огляду цільової архітектури.
- Використовуй [`project/architecture-spec.md`](project/architecture-spec.md) для компонентного дизайну, орієнтованого на реалізацію.
- Використовуй [`adr/README.md`](adr/README.md) для довготривалих архітектурних і процесних рішень.

## Рекомендації до написання

- Тримай технічну документацію українською в `docs_ua/`.
- Уникай дублювання одного й того самого факту в кількох файлах.
- Надавай перевагу посиланням на канонічні документи замість копіювання вмісту.
- Оновлюй [`../README.md`](../README.md) лише високорівневою інформацією про репозиторій; детальний внутрішній контекст тримай тут.
