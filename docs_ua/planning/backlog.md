# Беклог

Цей документ зберігає пріоритизовану роботу, яка ще не стала активною.

## Модель статусів

Для кожного елемента використовуй один із таких статусів:

- `Draft`
- `Ready`
- `Blocked`
- `Done`
- `Archived`

## Типи елементів планування

- `Epic`: великий елемент планування, орієнтований на результат, який об'єднує пов'язані `story`.
- `Story`: перевірюваний елемент постачання в межах одного `epic`, який можна декомпозувати на `task`.
- `Task`: найменший виконавчий елемент у межах однієї `story`, який описує конкретну роботу з реалізації.

Використовуй ієрархію `epic > story > task` під час декомпозиції.

## Угода про ієрархію в назві

Кодуй зв'язки з предками прямо в назві елемента, використовуючи `file stem` planning-файлів у квадратних дужках.

```md
- [Status] Epic title
- [Status][2026-03-14-epic-example-epic] Story title
- [Status][2026-03-14-epic-example-epic][2026-03-14-story-example-story] Task title
```

- Використовуй лише `file stem` предків без `.md` і без назв каталогів.
- Розташовуй сегменти в дужках від найвищого предка до найближчого.
- Не додавай у назву власний `file stem` поточного елемента.
- Якщо в предка ще немає окремого planning-файла, не додавай відповідний сегмент, доки файл не з'явиться.

## Шаблон елемента

Використовуй компактний формат для кожного елемента беклогу:

```md
- [Status][ancestor-file-stem-1][ancestor-file-stem-2] Коротка назва
  - Type: Epic | Story | Task
  - Чому це важливо:
  - Ознака прийняття:
  - Посилання:
```

## Правило супроводу

- Тримай список пріоритизованим зверху вниз.
- Підтримуй ієрархію в назвах відповідно до поточних назв planning-файлів, коли для пов'язаних предків уже існують файли.
- Переміщуй активні плани виконання в [`active/`](active/README.md).
- Переміщуй завершену або замінену роботу в [`archive/`](archive/README.md).
