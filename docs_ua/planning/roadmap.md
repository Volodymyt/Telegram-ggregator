# Дорожня карта

Цей документ відстежує теми постачання на ближню та середню перспективу.

Він має залишатися корисним як точка входу для декомпозиції беклогу, а не лише як вказівник на активні документи.

## Горизонт

Використовуй розділи, орієнтовані на віхи, наприклад:

- `Now`
- `Next`
- `Later`

## Рекомендації

- Формулюй елементи дорожньої карти через результат, а не через деталі реалізації.
- Посилайся на детальну роботу з виконання з [`active/`](active/README.md), коли елемент дорожньої карти стає активним.
- Переконайся, що кожна тема дорожньої карти все ще достатньо конкретна, щоб її можна було декомпозувати на віхи без повторного відкриття основних технічних рішень.
- Підтримуй трасованість тем дорожньої карти до продуктових вимог, критеріїв прийняття та активних планів постачання.
- Видаляй або переписуй застарілі теми замість накопичення неактуальних планів.

## Now

- Зробити постачання MVP готовим до декомпозиції, зафіксувавши послідовність віх, технічні значення за замовчуванням, семантику відновлення та покриття вимог для поточного базового стану реалізації. Детальний план: [2026-03-14-mvp-delivery-plan.md](active/2026-03-14-mvp-delivery-plan.md).
- Постачити перший виконуваний шлях MVP через foundations, intake, candidate creation, event deduplication і text publication з attribution. Детальний план: [2026-03-14-mvp-delivery-plan.md](active/2026-03-14-mvp-delivery-plan.md).

## Next

- Завершити повний життєвий цикл події, включно з обробкою `clear`, безпечним щодо перезапуску відновленням для станів candidate і publish, а також детермінованою логікою retry і failure handling під час publish. Просувати цю роботу лише після того, як перший end-to-end publish slice стане стабільним. Детальний план: [2026-03-14-mvp-delivery-plan.md](active/2026-03-14-mvp-delivery-plan.md).
- Довести сервіс до задокументованої планки прийняття MVP для media support, operability, capacity, latency і container hardening на цільовому VPS-профілі. Просувати цю роботу після того, як буде готове lifecycle recovery. Детальний план: [2026-03-14-mvp-delivery-plan.md](active/2026-03-14-mvp-delivery-plan.md).

## Later

- Перетворити перевірені висновки з MVP на пріоритизовані post-MVP елементи беклогу, не відкриваючи повторно прийняті архітектурні обмеження MVP і не втягуючи зарано позаскоупні можливості в активний потік. Беклог: [backlog.md](backlog.md).
