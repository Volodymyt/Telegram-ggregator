# ADR 0002: Використовувати Postgres для зберігання повідомлень і подій у MVP

## Статус

Прийнято

## Контекст

Сервіс має зберігати повідомлення з джерел, стан кандидатів, стан логічних подій і статус обробки між перезапусками.

Для MVP також потрібен простий спосіб:

- запобігати дубльованим репостам за допомогою довговічного унікального ключа,
- відстежувати, де кожне повідомлення з джерела перебуває в пайплайні,
- відстежувати, до якої логічної події належить кандидат,
- закривати й повторно відкривати логічні події між перезапусками,
- зберігати результати публікації та помилки,
- перевіряти стан під час операцій і налагодження.

Черги в пам'яті достатні для координації під час виконання, але вони не є довговічними сховищами. Тому шар зберігання має бути канонічним джерелом істини як для стану повідомлень, так і для стану логічних подій.

## Рішення

MVP використовуватиме Postgres як канонічний шар зберігання для повідомлень із джерел, стану кандидатів і стану логічних подій.

Під час виконання система буде:

- зберігати повідомлення з джерел і їхній статус обробки в Postgres,
- зберігати логічні події та їхній життєвий цикл у Postgres,
- використовувати унікальний ключ на `(source_chat_id, source_message_id)` для довговічної дедуплікації повідомлень із джерел,
- дозволяти агрегатору кандидатів відновлювати довговічні рядки `candidate` з Postgres після перезапуску,
- тримати черги обробки, кандидатів і публікації в процесі замість введення зовнішнього брокера,
- і далі зберігати Telegram-сесію як змонтований файл поза контейнерним образом.

## Наслідки

- Середовищу виконання потрібен рядок підключення до Postgres, наприклад `DATABASE_URL`.
- Документація має розглядати Postgres, а не SQLite, як канонічне сховище MVP.
- Оператори отримують сховище з можливістю запитів для статусів повідомлень, стану життєвого циклу подій, результатів публікації та збоїв.
- Стан черг лишається ефермерним, тому воркери й агрегатор кандидатів мають за потреби виводити відновлюваний стан із Postgres під час старту.
- Redis або інший брокер для MVP не потрібен.
