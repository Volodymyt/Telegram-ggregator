# Архітектура

Цей документ є коротким оглядом архітектури. Компонентний дизайн, орієнтований на реалізацію, описано в [`architecture-spec.md`](architecture-spec.md).

## Дизайн цільової системи

Цільовий MVP — це один асинхронний Python-сервіс, який:

- підключається до Telegram через збережену користувацьку сесію,
- використовує Telethon для підписки на події вихідних повідомлень,
- зберігає вихідні повідомлення, стан кандидатів і стан логічних подій у Postgres,
- фільтрує повідомлення у типізованих кандидатів на публікацію,
- агрегує кандидатів у логічні події з дедуплікацією на рівні події,
- публікує лише вибраних кандидатів подій в один цільовий канал через окремий внутрішній потік публікації.

## Основні компоненти

- Reader повідомлень: реєструє обробники Telethon для нових повідомлень, нормалізує вхідні події та записує щойно побачені повідомлення.
- Store повідомлень: зберігає вихідні повідомлення, стан кандидатів і стан логічних подій у Postgres.
- Потік обробки: використовує чергу в межах процесу й воркер для застосування типізованих правил include і exclude та позначення кандидатів на публікацію.
- Потік агрегації кандидатів: дедуплікує кандидатів між джерелами, відстежує життєвий цикл логічної події та вибирає канонічне джерело для публікації.
- Потік публікації: використовує окрему чергу в межах процесу й воркер для серіалізованої публікації в цільовий канал та повторних спроб.
- Спостережуваність: виводить структуровані логи й експонує статус здоров'я.

## Обмеження Telegram

- Telegram Bot API недостатньо для читання довільних зовнішніх вихідних каналів, якщо бота явно не додано з потрібними правами.
- Тому сервіс читає вихідні повідомлення через сесію Telegram-користувача по MTProto.
- Telethon володіє транспортом, перепідключеннями й підпискою на події. Застосунок реалізує лише обробку повідомлень і логіку пайплайну.

## Модель розгортання

- Один розгортний Python-сервіс у Docker.
- Змонтований файл Telegram-сесії для стану авторизації користувача.
- Postgres як канонічне сховище для повідомлень, кандидатів і стану подій.
- Файлова конфігурація разом зі змінними середовища для секретів і runtime-параметрів.

## Примітка про поточний репозиторій

- Поточний репозиторій усе ще містить мінімальний каркас-заглушку.
- `src/Telegram-aggregator/` слід вважати застарілим каркасом, а не цільовою структурою реалізації.
- Канонічним коренем пакета для нової реалізаційної роботи є `src/telegram_aggregator/`.
