# Вимоги проєкту

## Функціональні вимоги

### Джерела

- Джерела налаштовуються через файловий формат конфігурації та змінні середовища.
- Кожне джерело може бути задане як `@channel_username`, `t.me/...` або числовий ідентифікатор, де це підтримується.
- Сервіс має підтримувати щонайменше 100 каналів або чатів-джерел.
- Сервіс має підключатися до Telegram через Telethon, реєструвати обробники нових повідомлень і обробляти вхідні повідомлення майже в реальному часі.

### Фільтрація

- Фільтрація має підтримувати правила include та exclude.
- Правила include мають підтримувати regex-патерни й OR-вирази, розділені `|`.
- Кожне правило include має визначати `pattern`, `event_type` і `event_signal`.
- `event_signal` має підтримувати `start` і `clear`.
- Правила exclude мають підтримувати regex-патерни.
- Режим фільтрації має підтримувати `any` і `all`.
- Перевірка збігів має охоплювати і текст повідомлення, і підписи до медіа.
- Нормалізація має налаштовуватися й може включати перетворення до нижнього регістру, обрізання пробілів і нормалізацію символів на кшталт `ё` до `е`.
- У режимі `any` перше правило include, що збіглося в порядку конфігурації, визначає `event_type` і `event_signal` кандидата.
- У режимі `all` усі правила include мають мати однакові `event_type` і `event_signal`, інакше валідація конфігурації має завершитися помилкою.

### Поведінка репосту

- Кандидати `start`, вибрані агрегатором кандидатів, мають публікуватися в цільовий Telegram-канал.
- Текстові пости мають зберігати текст повідомлення.
- Медіапости мають по можливості зберігати медіа й підпис.
- Репости мають містити атрибуцію джерела та посилання на пост, якщо воно доступне.
- Якщо пересилання або копіювання обмежене, сервіс має дотримуватися налаштованої fallback-політики.
- Обробку альбомів у MVP можна звести до стратегії першого повідомлення.
- Кандидати `clear` мають закривати активну подію того самого `event_type` і не повинні публікуватися.

### Дедуплікація

- Сервіс має зберігати повідомлення з джерел, стан кандидатів і стан логічних подій у Postgres.
- Сервіс не повинен репостити те саме повідомлення більше одного разу.
- Основний ключ дедуплікації повідомлень із джерел — `(source_chat_id, message_id)`.
- Агрегація кандидатів також має запобігати публікації тієї самої логічної події більше одного разу між різними джерелами.
- Дедуплікація логічних подій має працювати в межах того самого `target_channel` і `event_type`.
- Агрегатор кандидатів має використовувати нечітку схожість нормалізованого вмісту кандидата, щоб зіставляти кандидатів із відкритими подіями.
- Реалізація схожості за замовчуванням має використовувати Python `difflib.SequenceMatcher` з порогом відношення `0.82`.
- Повторний кандидат `start`, що збігається більш ніж через 5 хвилин після початку початкової події, має відкривати нову подію.
- Канонічним джерелом для публікації має бути перший кандидат, який відкриває подію.
- Сигнал на основі посилання може посилювати схожість, якщо він доступний, але не повинен бути єдиною стратегією крос-джерельної дедуплікації.
- Стан дедуплікації має зберігатися між перезапусками.

### Стійкість

- Сервіс має автоматично перепідключатися після переривання сесії.
- Стани flood wait мають оброблятися очікуванням потрібної тривалості.
- Транзієнтні збої мають використовувати поведінку з backoff.
- Агрегатор кандидатів має відновлювати необроблені рядки `candidate` з Postgres після перезапуску.
- Публікація має серіалізуватися через внутрішню чергу з throttling, щоб зменшити тиск лімітів.

### Одноразова авторизація

- При першому старті без файла сесії сервіс має підтримувати одноразовий login-flow.
- Підтримувані операторські сценарії:
  - інтерактивний режим входу, керований через `LOGIN=1`, або
  - окрема команда входу на кшталт `python -m telegram_aggregator.login`.

## Нефункціональні вимоги

### Продуктивність

- Нормальна наскрізна затримка від появи повідомлення з джерела до репосту має лишатися в межах 10-30 секунд, коли flood waits не активні.
- Цільовий профіль розгортання - невеликий VPS із 1 vCPU і 512 MB RAM або більше.

### Надійність

- Після перезапуску сервіс має продовжувати роботу з відновленого стану дедуплікації.
- Логи мають бути структурованими: або JSON, або послідовний key-value формат.

### Безпека

- Telegram-секрети не мають бути вшиті в образ.
- `api_id`, `api_hash` і файл сесії мають надходити через змінні середовища й змонтоване сховище.
- Контейнер має підтримувати знижені привілеї та може опційно використовувати root filesystem тільки для читання.
- Пароль 2FA не повинен зберігатися у відкритому вигляді.

## Зовнішній контракт конфігурації

### Змінні середовища

- `TG_API_ID`: Telegram API ID.
- `TG_API_HASH`: Telegram API hash.
- `TG_SESSION_PATH`: шлях до збереженого файла сесії Telegram, наприклад `/data/session.session`.
- `DATABASE_URL`: рядок підключення до Postgres для зберігання повідомлень, кандидатів і стану подій.
- `TARGET_CHANNEL`: username або числовий ідентифікатор цільового каналу.
- `CONFIG_PATH`: шлях до файлової конфігурації сервісу.
- `LOG_LEVEL`: рівень логування runtime, наприклад `INFO`, `DEBUG` або `WARNING`.
- `DRY_RUN`: необов'язковий режим, що логує збіги без публікації.
- `LOGIN`: необов'язковий режим, що запускає одноразову інтерактивну авторизацію.

### Файл конфігурації

Цільова форма конфігурації така:

```yaml
sources:
  - "@channel1"
  - "@channel2"

filters:
  mode: any
  include:
    - pattern: "Київ|Спуск|Балістика"
      event_type: ballistic
      event_signal: start
    - pattern: "чисто"
      event_type: ballistic
      event_signal: clear
  exclude:
    - "реклама|підписуйтесь"
  case_insensitive: true
  normalize: true

repost:
  add_source_footer: true
  footer_template: "Джерело: {source}\n{link}"
  fallback_on_copy_forbidden: "link_only"

runtime:
  processing_queue_size: 1000
  candidate_queue_size: 1000
  publish_queue_size: 200
  candidate_similarity_threshold: 0.82
  event_reopen_window_seconds: 300
  candidate_recovery_scan_seconds: 15
```

## Тестування та прийняття

### Unit-покриття

- Логіка фільтрів має покривати режими `any` і `all`.
- Правила exclude мають блокувати повідомлення, що збігаються.
- Потрібно перевіряти поведінку `case_insensitive` і нормалізації.
- Дедуплікація повідомлень із джерел має відхиляти повторну обробку того самого повідомлення.
- Агрегація кандидатів має придушувати дублікати логічних подій між кількома джерелами.
- Обробка `clear` має закривати лише активну подію того самого `event_type`.
- Restart recovery має повторно обробляти збережених кандидатів, яких ще не було агреговано.

### Сценарій ручної інтеграції

- Given two source channels and a filter that matches `Київ|Спуск|Балістика`
- When one source publishes a matching `start` message and another source publishes a similar message within 5 minutes
- Then the target channel receives a repost with attribution
- And only the first matching source is published
- And restarting the container does not create a duplicate repost

- Given an active logical event for `event_type=ballistic`
- When a matching `clear` message arrives
- Then the event is closed in persistence
- And no new target-channel post is created

### Критерії прийняття MVP

- Сервіс працює щонайменше 24 години без ручного втручання.
- Сервіс републікує принаймні текстові повідомлення й фото з підписами.
- Дедуплікація працює між перезапусками і між кількома джерелами для тієї самої події.
- Усі параметри runtime надходять зі змінних середовища та конфігурації.
- Поставка містить Docker-образ і інструкції запуску.
